# 分類

この分類は、CEHモジュール、試験ドメイン、CEH+実践トラック、質問メタデータ、分析、およびレビューのルーティングのための分類ソースです。ランタイム定数は `src/data/taxonomy.ts` に存在し、質問の形状は [QUESTION_SCHEMA.md](QUESTION_SCHEMA.md) に文書化されています。

## 質問分類フィールド

| フィールド | 必須 | 意味 | ソース |
|---|---:|---|---|
| `module` | はい | CEH知識モジュールのための `1`-`20`、またはCEH+実践アイテムのための `0`。 | 質問スキーマとバリデーター。 |
| `track` | 条件付き | `module` が `0` の場合のみ必須。 | 以下のCEH+トラックリスト。 |
| `difficulty` | はい | `easy`、`medium`、または `hard`。 | 質問スキーマとエディター。 |
| `tags` | はい | `scope`、`iam`、`logs`、または `reporting` などの検索可能なサブトピックとスキルヒント。 | シードJSON、エディター、およびインポートバリデーター。 |
| `type` | はい | `mcq`、`multi`、`true_false`、`short_answer`、`scenario`、または `report_prompt`。 | 質問スキーマとレンダラー。 |
| `source` | 任意 | `seed` または `user`；シード行が欠落している場合は `seed` に正規化され、インポートは `user` に正規化されます。 | ローダーと質問パックパーサー。 |

`tags` は、サブトピックとスキルタイプの現在の軽量表現です。将来のスキーマでは、モジュールやトラックの分類を変更することなく、これらをファーストクラスの `subtopic` と `skill_type` フィールドに分割できます。

## CEH+実践トラック

CEH+アイテムは `module: 0` を使用し、これらのトラックキーのいずれかを使用します。

| トラックキー | トラック | 安全な焦点 |
|---|---|---|
| `pentest` | ペンテストエンゲージメントワークフロー | スコープ、エンゲージメントのルール、証拠、発見のトリアージ、および報告の構造。 |
| `appsec` | アプリケーションセキュリティコードレビュー | 認証、入力検証、秘密情報の取り扱い、依存関係リスク、および安全な修正の理由。 |
| `cloud` | クラウドIAM / 設定レビュー | 最小権限、公開露出、ログ記録、暗号化、および秘密情報の取り扱い。 |
| `soc` | SOCログ調査 | タイムライン、指標、影響を受けた資産、封じ込め、およびエスカレーション。 |
| `ir` | インシデントレスポンス | 証拠の保存、封じ込め、根絶、回復、および教訓。 |
| `threat-model` | 脅威モデリングと修正 | 資産、信頼境界、STRIDEスタイルの脅威、緩和策、および修正計画。 |

報告は、別のトラックキーではなく、Safe LabsとReportsを通じて実装されたクロストラックワークフローです。トラックには、報告準備が整った証拠、影響、および修正の実践が含まれるべきです。

## モジュールからCEH+へのマッピング

このマッピングは、ドリルの選択とコンテンツの作成をガイドします。公式のCEHモジュールリストの代替ではありません。

| CEHエリア | 関連モジュール | 関連CEH+トラック |
|---|---|---|
| 倫理、スコープ、方法論 | 1 | `pentest`, `ir` |
| 偵察と資産発見の概念 | 2, 3, 4 | `pentest`, `soc` |
| 脆弱性とホストリスクの推論 | 5, 6, 7 | `pentest`, `ir`, `threat-model` |
| ネットワーク、周辺、ソーシャル、セッションリスク | 8, 9, 10, 11, 12 | `soc`, `ir`, `threat-model` |
| ウェブサーバー、ウェブアプリ、インジェクションリスク | 13, 14, 15 | `appsec`, `pentest` |
| 無線、モバイル、IoT、OTリスク | 16, 17, 18 | `soc`, `threat-model` |
| クラウドと暗号学 | 19, 20 | `cloud`, `appsec`, `threat-model` |

## 分類されていない検出ルール

シードバリデーションと質問パックのインポートは、分類できない行を拒否または正規化しなければなりません：

- `module` は `0` から `20` の整数でなければなりません。
- `module: 0` には有効なCEH+ `track` が必要です。
- `module: 1..20` はCEH+ `track` に依存してはなりません；ローダーはモジュールからドメインと地区を導出します。
- `difficulty`、`type`、`tags`、`body`、`answer`、および説明フィールドはスキーマバリデーションを通過しなければなりません。
- シードファイルまたは単一の質問パック内の重複IDは拒否されます。

シード編集後に `npm run validate:content` を実行してください。
