# コンテンツ作成ガイド

このガイドは、質問とラボのリリース安全な作成について説明します。すべてのコンテンツは、[SAFETY_BOUNDARIES.md](SAFETY_BOUNDARIES.md)に従う必要があります：認可された防御学習、合成/ローカルアーティファクトのみ、実際のターゲット、実際の資格情報、運用攻撃手順は含まれません。

## 真実の源

- シード質問: `src/data/questions/group-*.json`
- 概念カード: `src/data/conceptCards.ts`
- アプリ内カスタム質問: ブラウザストレージ、設定から質問パックとしてエクスポート可能
- 安全なラボ: `src/data/labs.ts`
- 質問検証スクリプト: `npm run validate:content`
- 質問スキーマリファレンス: [QUESTION_SCHEMA.md](QUESTION_SCHEMA.md)
- インポート/エクスポートリファレンス: [IMPORT_EXPORT.md](IMPORT_EXPORT.md)

## 質問作成

各質問は1つの概念をテストするように作成します。学習者は正しい答え、その理由、次回避けるべき罠、短い記憶フレーズを持ち帰るべきです。

すべての質問に必要なフィールド：

| フィールド | メモ |
|---|---|
| `id` | 安定した一意のID。シードIDは再利用しないでください。ユーザーパックIDの衝突はインポート時に名前が変更されます。 |
| `title` | 短いスタディカードのタイトル。ユーザー質問のアプリ内作成フォームでは必須ですが、古いシード/インポート行ではオプションです。 |
| `type` | `mcq`、`multi`、`true_false`、`short_answer`、`scenario`、または`report_prompt`。 |
| `module` | CEHモジュール `1`-`20`、またはCEH+実践トラックの場合は`0`。 |
| `track` | `module`が`0`の場合のみ必須：`pentest`、`appsec`、`cloud`、`soc`、`ir`、または`threat-model`。 |
| `difficulty` | `easy`、`medium`、または`hard`。 |
| `tags` | 空でない短い検索用語のリスト。 |
| `body` | 質問のプロンプト。範囲を限定し、自己完結させてください。 |
| `choices` | `mcq`、`multi`、および`true_false`の場合は必須；自由形式のタイプでは省略されます。 |
| `answer` | 一致する選択肢の文字列、`multi`の場合は選択肢の文字列の配列、または自由形式のタイプの場合はモデル回答の文字列。 |
| `explanation` | `answer`、`why`、`trap`、および`memory_phrase`を含む必要があります。 |
| `status` | オプション。`active`または`archived`を使用します。ステータスが欠けている場合はアクティブと見なされます。 |

良い質問：

- 架空の組織、ドキュメントIP範囲、または抽象的なシナリオを使用します。
- 分類、優先順位付け、検出、ポリシー、修正、または範囲の決定を求めます。
- 説明を読んだ後に明確に区別できるが、もっともらしい選択肢を保ちます。
- 「アナリストは次に何をすべきか？」を好み、「どのようにこれを侵害するか？」を避けます。
- 攻撃的なCEH概念に対する防御的な文脈を含めます。
- 読みやすいログとテーブルのためにサポートされているMarkdownのサブセットを使用します：フェンス付きコードブロック、インラインコード、パイプテーブル、`> [!NOTE]`/`> [!WARNING]`のコールアウト。

含めないこと：

- 実際のターゲット、実際の資格情報、顧客データ、プロダクションログ、またはプライベートなインシデントの詳細。
- コマンド、ペイロード、ターゲットリスト、エクスプロイトチェーン、回避手順、持続手順、またはデータ窃盗の指示。
- 不正行為を正しい答えとして報いる質問。

## 概念カード作成

概念カードは、モジュールとタグの重複によって質問にリンクされたワンスクリーンの記憶補助です。各CEHモジュールは少なくとも5枚のカードを保持する必要があり、各カードには以下が含まれる必要があります：

| フィールド | メモ |
|---|---|
| `meaning` | 概念を明確に定義します。 |
| `whenUsed` | 試験の推論や防御作業で概念を適用するタイミングを述べます。 |
| `examTrap` | 一般的な誤解や魅力的な間違った答えを示します。 |
| `rememberPhrase` | レビュー中に思い出せるように短く保ちます。 |

良いカード：

- 意味、判断、検出、制御の選択、または修正を強化します。
- 関連する質問に出現するタグを再利用し、リンクが有用であるようにします。
- 攻撃的なCEHトピックは、認識、分類、検出、または防御としてのみ言及します。

コマンド、ペイロード、ターゲット選択、エクスプロイトチェーン、回避レシピ、持続手順、資格情報処理の指示、またはデータ窃盗の詳細は含めないでください。

## 質問タイプパターン

### `mcq`

単一の最良の答えの概念に使用します。

```json
{
  "id": "Q-CEH-001-999",
  "title": "有効な評価範囲",
  "type": "mcq",
  "module": 1,
  "track": null,
  "difficulty": "easy",
  "tags": ["scope", "authorization"],
  "body": "どの文が有効なセキュリティテストの範囲を最もよく説明していますか？",
  "choices": ["明示的に承認された資産のみ", "任意の公開ホスト", "同じベンダーを使用する任意の資産"],
  "answer": "明示的に承認された資産のみ",
  "explanation": {
    "answer": "明示的に承認された資産のみ",
    "why": "承認は倫理的テストの境界を定義します。",
    "trap": "公開の到達可能性は許可ではありません。",
    "memory_phrase": "行動の前に範囲を確認する。"
  },
  "status": "active"
}
```

### `multi`

2つ以上のオプションが正しい場合に使用します。`answer`の値は配列であり、すべての項目は選択肢と正確に一致する必要があります。

### `true_false`

正確にこれらの選択肢を使用します：

```json
"choices": ["True", "False"]
```

`answer`の値は`"True"`または`"False"`でなければなりません。

### `short_answer`

簡潔な自己評価のために使用します。`answer`の値はモデル回答の文字列です。

### `scenario`

短いモデル回答のプロンプトに使用します。シナリオ質問は選択質問として自動採点されず、推論、報告言語、安全な実践トラックに役立ちます。

### `report_prompt`

合成された発見や報告作成の練習に使用します。プロンプトはタイトル、影響、証拠、修正、または範囲の言語を求め、モデル回答は提供された架空または合成の証拠の範囲内に留まるべきです。

## 説明パターン

```text
Answer:
正しい答えを明確に述べます。

Why:
理由と防御の原則を説明します。

Trap:
誤解や魅力的な間違った答えを示します。

Memory phrase:
学習者がレビュー中に思い出せる短いフレーズ。
```

## ラボ作成

安全なラボは静的で合成された読み取り専用の演習です。これらは`src/data/labs.ts`に保管され、学習者が提供された証拠と報告ビルダーを使用してアプリ内でラボを完全に完了できるように作成されるべきです。

各ラボには以下が必要です：

| フィールド | メモ |
|---|---|
| `id` | ルート/ローカル進行キーで使用される安定したスラッグ。 |
| `title` | ユーザー向けのラボタイトル。 |
| `category` | `SOC`、`Cloud`、`Web`、`Network`、`Social`、または`Threat Model`などの短いグループ。 |
| `kind` | `local`、`dataset`、`simulated`、または`writeup`のいずれか。 |
| `difficulty` | `easy`、`medium`、または`hard`。 |
| `brief` | 学習者が分析している内容と期待される成果物。 |
| `scope.allowed` | 学習者が使用できる正確なローカル/合成アーティファクトとアプリツール。 |
| `scope.forbidden` | 実際のシステム、外部検索、トラフィック生成、資格情報、ペイロード実行を明示的に禁止します。 |
| `evidenceTitle` / `evidence` | 静的な合成アーティファクト。架空の名前とドキュメントIP範囲を使用します。 |
| `flagChallenge` | プロンプト、ローカルアセットメタデータ、期待されるユニークなフラグ、範囲のヒント、説明、修正、報告プロンプト。 |
| `objectives` | スコアリングコンポーネントにマッピングされたチェックリスト項目。 |
| `rubric` | フラグ/診断、証拠、説明、修正、安全性のためのコンポーネントスコアリング。 |
| `guiding` | 教育的な回答を含むヒント。ヒントは実世界の行動手順を導入してはいけません。 |
| `modelFindings` | 報告の引き渡しのための重大度、影響、修正を含む発見。 |

## ラボチェックリスト

- [ ] ラボはアプリを離れることなく完了できます。
- [ ] スコープ契約が表示され、具体的です。
- [ ] 証拠は架空または合成であり、実際の秘密や個人データを含みません。
- [ ] 期待されるフラグはユニークな`FLAG{UPPER_SNAKE_CASE}`値を使用し、ローカルトレーニングの回答のみです。
- [ ] すべてのフラグチャレンジアセットは、アプリ内で既に提供された資料を説明します。
- [ ] 禁止リストは実際のターゲット、資格情報、サードパーティサービス、トラフィック生成、ペイロード実行を除外します。
- [ ] 目的は証拠と修正を必要とし、単に脆弱性を名前付けするだけではありません。
- [ ] ヒントは操作的な誤用手順を与えずに推論を教えます。
- [ ] 説明、修正、報告プロンプトは防御的であり、受け入れられたフラグの後に安全に解除されます。
- [ ] モデルの発見は報告準備が整っており、合成の範囲言語を使用します。
- [ ] 報告ビルダーはラボのタイトル、範囲、要約、モデルの発見を安全に使用できます。
- [ ] `npm run validate:safety`が公開前に合格します。

## レビューチェックリスト

- [ ] 質問または目的ごとに1つの概念。
- [ ] 正しいCEHモジュールまたはCEH+トラック。
- [ ] 難易度が期待される学習者の努力に一致します。
- [ ] 説明には答え、理由、罠、記憶フレーズが含まれています。
- [ ] 実際のターゲット、資格情報、秘密、個人データ、またはプロダクションアーティファクトはありません。
- [ ] 操作的な攻撃チェーン、回避、持続、窃盗、または悪用の指示はありません。
- [ ] シード質問JSONは`npm run validate:content`に合格します。
- [ ] インポートされた/ユーザー作成のパックは[IMPORT_EXPORT.md](IMPORT_EXPORT.md)に従います。
