# 質問スキーマ

これは、シードコンテンツバリデーターおよび質問パックインポートパーサーによって使用されるリリーススキーマです。著作ガイダンスについては[CONTENT_GUIDE.md](CONTENT_GUIDE.md)を、必須の安全ルールについては[SAFETY_BOUNDARIES.md](SAFETY_BOUNDARIES.md)を参照してください。

## 必須フィールド

| フィールド | タイプ | 必須 | ノート |
|---|---|---:|---|
| `id` | 文字列 | はい | 安定した一意のID。 |
| `title` | 文字列 | いいえ | 短いユーザー向けタイトル。ユーザー質問のアプリ内著作フォームで必須; レガシーシード行およびインポートではオプション。 |
| `type` | 文字列 | はい | `mcq`、`multi`、`true_false`、`short_answer`、`scenario`、または`report_prompt`。 |
| `module` | 数字 | はい | CEHモジュールの`1`-`20`、CEH+トラックの`0`。 |
| `track` | 文字列またはnull | 条件付き | `module`が`0`のときに必須; それ以外の場合は`null`を使用。 |
| `difficulty` | 文字列 | はい | `easy`、`medium`、または`hard`。 |
| `tags` | 文字列[] | はい | 空であってはならない。 |
| `body` | 文字列 | はい | プロンプトテキスト。 |
| `choices` | 文字列[] | タイプによる | `mcq`、`multi`、および`true_false`の場合は必須; 自由形式タイプでは省略。 |
| `answer` | 文字列または文字列[] | はい | 以下のタイプ特有のルールに一致しなければならない。 |
| `explanation.answer` | 文字列 | はい | 単純な回答の要約。 |
| `explanation.why` | 文字列 | はい | 理由と防御原則。 |
| `explanation.trap` | 文字列 | はい | 一般的な誤解または間違った道。 |
| `explanation.memory_phrase` | 文字列 | はい | 短いレビューのフレーズ。 |
| `status` | 文字列 | いいえ | `active`または`archived`; 欠落はアクティブを意味する。 |
| `source` | 文字列 | いいえ | `seed`または`user`; インポートは`user`に正規化される。 |

## サポートされているMarkdown

質問の本文、選択肢、説明、およびモデル回答は制限されたMarkdownのサブセットをレンダリングします：

- 段落、順不同リスト、順序付きリスト、`inline code`、`**bold**`、および`*emphasis*`
- コピーボタン付きのフェンコードブロック
- ヘッダーセパレータ行を持つパイプテーブル
- `> [!NOTE] Synthetic evidence`のようなブロック引用構文を使用したコールアウト

生のHTMLはレンダリング前にエスケープされます。スクリプトタグ、画像、iframe、リモートアセット、またはインラインHTML属性に依存しないでください。

## サポートされている質問タイプ

### `mcq`

- `choices`には少なくとも2つの文字列が含まれている必要があります。
- `answer`は、正確に1つの選択肢と一致する1つの文字列でなければなりません。

### `multi`

- `choices`には少なくとも3つの文字列が含まれている必要があります。
- `answer`は少なくとも2つの文字列を含む配列でなければなりません。
- すべての回答文字列は正確に選択肢と一致しなければなりません。

### `true_false`

- `choices`は正確に`["True", "False"]`でなければなりません。
- `answer`は`"True"`または`"False"`でなければなりません。

### `short_answer`

- `choices`は省略されます。
- `answer`は簡潔なモデル回答の文字列です。
- 学習者は自分のテキストをモデル回答と比較した後に自己評価を行います。

### `scenario`

- `choices`は省略されます。
- `answer`はモデル回答の文字列です。

### `report_prompt`

- `choices`は省略されます。
- `answer`はモデルレポートまたは発見の回答文字列です。
- 合成レポート作成、証拠、影響、および修正の練習に使用します。

## CEH+トラック

`module`が`0`のとき、`track`は次のいずれかでなければなりません：

- `pentest`
- `appsec`
- `cloud`
- `soc`
- `ir`
- `threat-model`

## コンセプトカードスキーマ

コンセプトカードは`src/data/conceptCards.ts`で著作され、質問バンクの安全な学習仲間です。手続き的ではなく、概念的かつ防御的である必要があります。

| フィールド | タイプ | 必須 | ノート |
|---|---|---:|---|
| `id` | 文字列 | はい | 安定した`CC-CEH-MM-NN`識別子。 |
| `module` | 数字 | はい | CEHモジュール`1`-`20`。 |
| `title` | 文字列 | はい | 短い概念名。 |
| `tags` | 文字列[] | はい | 相互リンクのために質問タグと重複する必要があります。 |
| `meaning` | 文字列 | はい | 概念の意味。 |
| `whenUsed` | 文字列 | はい | 学習者が概念を適用すべき時。 |
| `examTrap` | 文字列 | はい | 一般的な間違ったメンタルモデルまたは回答の罠。 |
| `rememberPhrase` | 文字列 | はい | 短いリコールフレーズ。 |

すべてのCEHモジュールには少なくとも5つのコンセプトカードが必要です。カードビューワリンクは、モジュールと重複するタグに基づいて生成されます; 実際のターゲット、コマンド、ペイロード、エクスプロイトチェーン、回避手順、持続手順、資格情報処理の指示、またはデータ窃盗の詳細を避けてください。

## バリデーション

シード質問JSONを編集した後、シードバリデーターを実行します：

```bash
npm run validate:content
```

質問パックのインポートは同じコアルールを使用し、形式が不正なJSON、サポートされていない形式/バージョン、不正な質問フィールド、パック内の重複ID、および選択肢と一致しない回答を拒否します。既存のローカルIDの衝突は、確認されたパックインポート中に自動的に名前が変更されます。

## 品質ルール

- 1つの質問は1つの概念をテストします。
- すべての説明は回答、理由、罠、および記憶フレーズを述べます。
- 誘導選択肢はもっともらしいが曖昧ではありません。
- 攻撃的なCEH概念は、認識、検出、範囲、予防、または修正としてフレーミングされます。
- 実際のターゲット、資格情報、秘密、個人データ、運用攻撃チェーン、回避手順、持続手順、または情報漏洩の指示はありません。
