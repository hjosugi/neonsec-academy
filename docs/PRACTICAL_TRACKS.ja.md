# CEH+ 実践トラック

CEH の先を見据え、次の実践トラックを用意する。

## トラック 1: ペンテストエンゲージメントワークフロー

焦点: スコープ、エンゲージメントのルール、証拠、発見のトリアージ、報告。

セーフモード: 合成組織と静的データセットのみ。

## トラック 2: アプリケーションセキュリティコードレビュー

焦点: 認可、入力検証、秘密情報の取り扱い、依存関係リスク、セキュアな修正。

セーフモード: おもちゃのコードスニペットのみ。

## トラック 3: クラウド IAM / 設定レビュー

焦点: 最小権限、公開露出、ログ記録、暗号化、秘密情報の取り扱い。

セーフモード: 合成設定ファイルのみ。

## トラック 4: SOC ログ調査

焦点: タイムライン、インジケーター、影響を受けた資産、封じ込めのアイデア。

セーフモード: 合成ログのみ。

## トラック 5: インシデントレスポンス報告

焦点: タイムライン、影響、封じ込め、根絶、回復、教訓。

セーフモード: 架空のインシデントシナリオのみ。

## トラック 6: 脅威モデリングと修正

焦点: 資産、信頼境界、脅威、緩和策。

セーフモード: 架空のアーキテクチャ図のみ。

## ラボスコアリングルブリック

セーフラボは、以下のコンポーネントで実践作業をスコアリングします：

- フラグ / 診断
- 証拠
- 説明
- 修正
- 安全性の確認

各ラボはチャレンジタイプを定義し、ルブリックコンポーネントをその目的にマッピングします。ヒントと欠落したスコープの確認はペナルティを適用します。設定により、合格スコア、ヒントペナルティ、およびスコープ警告ペナルティを調整できますが、ラボの内容は変更されません。

フラグコンポーネントは、受け入れられた `FLAG{UPPER_SNAKE_CASE}` の提出によってのみ獲得されます。試行とユニークなヒントの公開はローカルに保持され、分析に供給されます。受け入れられたフラグはさらなる提出をロックし、説明/修正/報告のプロンプトを解除し、モデルの発見を有効にします。ガイディングクエスチョンのヒントとフラグチャレンジのヒントは、設定されたヒントペナルティに寄与します。

## ラボレジストリの安全性

すべてのラボは `kind`: `local`, `dataset`, `simulated`, または `writeup` を宣言します。ラボの詳細は証拠の前にスコープ契約を表示し、学習者は開始前に合成の読み取り専用境界を確認する必要があります。レジストリバリデーターは、欠落したスコープ、無効な種類、公開IPアドレス、実際のメールドメイン、ライブドメイン、および資格情報のような割り当てを拒否します。CIはまた、`npm run validate:safety` を実行し、安全でないラボメタデータが公開されないようにします。レジストリの検証には、プロンプト、ローカル資産メタデータ、ユニークな期待されるフラグ、少なくとも1つのスコープ付きヒント、説明、修正、および各ラボの報告プロンプトが必要です。
